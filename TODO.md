# Fully Self-Contained `examples/add.py` Plan

Goal: make `examples/add.py` run the NV add kernel end to end without importing tinygrad. The only acceptable external boundary is OS/hardware access: Linux NVIDIA ioctls or macOS eGPU/TinyGPU PCIe transport.

## Current State

- `examples/add.py` builds the cubin locally.
- `examples/add.py` locally parses ELF sections, applies CUDA relocations, builds cbuf0, writes QMD bitfields, submits GPFIFO words, rings the doorbell, waits on the timeline, and copies out the result.
- `examples/add.py` no longer has an active `nv_pcie`/tinygrad transport path; standalone transports are selected directly.
- obsolete `nv_pcie.py` wrapper has been deleted.
- GSP queue/WPR/Falcon boot scaffolding is local; `NV_ADD_BOOT_GSP=1` attempts the Ampere/Ada boot path and waits for `GSP_INIT_DONE`.
- GSP RM control responses now parse and validate the embedded RM status field before returning payload bytes; selftests cover success and error responses.
- GSP `ALLOC_MEMORY` packing now accepts both raw physical page lists and `(base, span)` ranges, validates that enough pages back the requested length, and selftests cover the packed page descriptors.
- The standalone backend facade now delegates through the same high-level API, and the booted GSP path attempts RM base-object allocation, handle-backed sysmem, compute GPFIFO allocation, scheduling, and initial compute queue setup.
- The public `NvBackend` facade API from Milestone 1 is now pinned by offline selftest with a fake inner device: device/interface names, memory-window properties, allocation/copy/free/sync/close, kernel-arg allocation, timeline state, GPFIFO submission, timeline signaling, wait, and context-manager close all forward through the facade without opening hardware. The simulated backend facade is checked against the same contract.
- The booted GSP path now explicitly allocates the fixed RM root object before device/subdevice/vaspace/channel allocations.
- The standalone launch path now has explicit wait/cache-invalidate/PCAS launch packets and programs shader local memory backing before kernels that request it.
- The Linux ioctl transport now discovers the NVIDIA PCI function through sysfs and can read/write PCI config space plus report/map BAR resources; the active `sys.path` insertion has been removed.
- The macOS/TinyGPU transport now exposes `PROBE`, `PING`, `RESET`, and `RESIZE_BAR` protocol commands, with selftests validating command id, BAR index, and requested size packing through the shared RPC wrapper.
- The TinyGPU MMIO view now has selftest coverage for typed and byte-slice BAR reads/writes, including command id, BAR index, offset, and little-endian payload packing.
- TinyGPU sysmem header parsing is now factored and selftested; `(paddr, span)` pairs are expanded into 4KB physical pages and truncated to the requested allocation size.
- The Linux ioctl transport now has direct wrappers for `NV_ESC_CARD_INFO`, `NV_ESC_REGISTER_FD`, `NV_ESC_RM_ALLOC`, and `NV_ESC_RM_CONTROL`, with selftests validating ioctl request construction and RM control parameter copyback.
- The Milestone 3 transport abstraction is now pinned by a no-hardware `--transport-contract` check in both examples: it verifies the required `Transport` surface (`read_config`, `write_config`, `bar_info`, `map_bar`, `alloc_sysmem`, `sleep`) with a fake transport, including typed BAR views, rounded sysmem page lists, and call forwarding.
- The Linux ioctl transport now has a direct `NV_ESC_RM_MAP_MEMORY` wrapper using the local NVOS33 packer; selftests validate map-memory request packing and returned linear-address copyback.
- The Linux ioctl transport now has a direct `NV_ESC_RM_MAP_MEMORY_DMA` wrapper using the local NVOS46 packer, and the Linux sysmem path uses that wrapper; selftests validate DMA map request packing and status handling.
- A `LinuxRmClient` adapter now exposes the common RM client API on top of Linux ioctls, so the booted Linux path can reuse channel-builder allocation code; selftests cover root allocation, object allocation, and handle-backed sysmem allocation through the adapter.
- The Linux booted-GSP path now fetches the binary GPU UUID through RM control, registers the GPU and VA space with UVM, and configures Linux sysmem allocation handles after RM base object allocation.
- The Linux path now registers the compute GPFIFO channel with UVM after channel creation.
- Linux UVM create-external-range, map-external-allocation, and free helpers now validate returned status fields themselves; `alloc_sysmem()` routes through the map helper, and selftests cover request packing for all three helpers.
- Linux UVM channel registrations now have an explicit range-release helper, and standalone backend close releases the registered UVM channel before RM unload. Backend close now attempts UVM channel release, runtime-buffer release, and RM unload even if an earlier cleanup stage fails, then raises the first cleanup error; selftests cover channel VA recycling, failure-tolerant teardown, and idempotent close cleanup.
- Linux UVM channel registration now unwinds failed registrations by recycling the reserved channel VA range, and booted resource setup releases allocated runtime buffers if late UVM channel registration fails. Selftests cover status-failure VA reuse plus late booted-resource cleanup.
- Linux sysmem allocations now keep enough metadata to release UVM external ranges, unmap fixed CPU mappings, and recycle virtual address ranges; cleanup clears released metadata so repeated frees are no-ops, with selftest coverage.
- The standalone backend now exposes explicit `free()` and idempotent `close()` lifecycle hooks; booted-GSP close sends `UNLOADING_GUEST_DRIVER`, and selftests cover sysmem virtual-address recycling.
- Standalone buffer frees are now idempotent even when called on offset views; freed base buffers clear their CPU view/metadata, stale offset views reject later CPU access, and repeated public `free()` calls become no-ops, with selftest coverage.
- Standalone sysmem allocation now validates that transports return enough 4KB physical pages before mapping, and selftests cover a short-page-list transport failure.
- Standalone backend close now releases backend-owned runtime buffers (`gpfifo_area`, `cmdq_page`, `kernargs_buf`, timeline signal, and shader local memory) through the allocator before RM unload. Runtime-buffer release is best-effort across context-mapping cleanup and each buffer, so malformed retained context metadata still reports an error but does not prevent ordinary buffer cleanup; selftests cover resource release, malformed-mapping cleanup continuation, and repeated close.
- Booted channel notifier memory is now retained in `ChannelResources` and released with the rest of the backend-owned runtime buffers; selftests cover RM-channel notifier ownership and VA reuse after close.
- The standalone backend now has a named semaphore-only timeline signal path; selftests validate its method packets, GPFIFO ring submission, and simulated timeline completion before the hardware boundary. Both examples now expose `--channel-contract`, a no-hardware Milestone 10 check that builds simulated runtime channel resources, verifies GPFIFO ring/control/token pages, submits semaphore-only timeline signals through the public facade, checks command-queue placement plus BAR0 doorbell token writes, validates compute-queue setup packets, and exercises shader-local-memory programming. ## Risks And Notes

- GSP boot and RM structs are the largest risk; copying too much generated ABI will make `add.py` unreadable.
- Prefer compact hand-written ctypes structs for only the commands used by add.
- Keep Blackwell support out until Ampere/Ada add works end to end.
- Do not add compatibility layers for unused features like NVDEC, PMA profiling, RDMA, peer access, or multi-GPU.
- The macOS eGPU path needs TinyGPU for physical sysmem pages and BAR access; raw ioctls are not available there.
- LibreCuda is useful for QMD/ioctl behavior, but this repo currently does not vendor LibreCuda code.
