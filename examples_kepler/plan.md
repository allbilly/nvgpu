# GK104 A0C0 `SET_OBJECT` Debug-and-Fix Plan

## 1. Current proven boundary

The current test establishes all of the following:

* PBDMA fetches and consumes the GPFIFO entry and pushbuffer.
* The pushbuffer reaches PGRAPH.
* The failure occurs on the first GR method:

  ```text
  SET_OBJECT subchannel=1, class=0xA0C0
  ```
* No QMD address, descriptor method, launch method, SASS instruction, or user-data access is required to reproduce the trap.
* An empty genuine `sm_30` kernel gives the same failure.
* Trap registers are clean immediately before submission.
* The semaphore remains unchanged because its WFI release is ordered behind the faulted GR operation.

The current packet generated by:

```python
nvm(1, 0x0000, KEPLER_COMPUTE_A)
```

should be exactly:

```text
word 0: 0x20012000    type=INC, count=1, subchannel=1, method=0
word 1: 0x0000a0c0    KEPLER_COMPUTE_A
```

The first assertion in every test run should verify those exact two words in GPU-visible memory. The repository itself defines `0xA0C0` as `KEPLER_COMPUTE_A`, and Nouveau’s GK104 GR class list explicitly supports that class.

## 2. Ranked working hypotheses

### Highest probability: malformed or incomplete GR context restore

`SET_OBJECT` is the first method routed to GR. It can therefore trigger channel context activation. The trap appearing in multiple GPCs and specifically in MP/TPC units is much more suggestive of invalid restored state than of a missing class lookup.

The suspicious topology pattern is:

```text
GPC1/TPC1  traps
GPC2/TPC1  traps
GPC3/TPC1  traps
```

That pattern should focus attention on:

* GPC/TPC replication loops;
* incorrect GPC or TPC register stride;
* a missing second-TPC initialization;
* an incomplete context pack;
* malformed class-specific method state;
* a golden context saved before every GPC/TPC was initialized.

The successful byte-for-byte copy verification proves that the runtime context matches the golden image. It does **not** prove that the golden image is semantically correct.

### Second: malformed per-channel MMIO-list binding

The runtime context header references the per-channel MMIO list used for pagepool, bundle, attribute and LTC state. An incorrect:

* count;
* list address;
* address shift;
* PTE target;
* privileged PTE flag;
* entry value;
* entry ordering;

could fail as soon as FECS restores the channel.

Nouveau’s `gf100_gr_chan_bind()` copies the golden context and then patches the channel-specific MMIO-list count and `VA >> 8` into the context header.

### Third: incorrect GR-context VMA attachment

On GK104, Nouveau writes the mapped GR-context VMA into the channel instance at engine offset `0x210`:

```text
RAMIN + 0x210 = low32(gr_context_va) | 0x4
RAMIN + 0x214 = high32(gr_context_va)
```

A physical address, stale virtual address, incorrect address space, missing `| 0x4`, or bad page-table mapping would produce a failure at first GR use.

### Fourth: GPU-side stale or corrupted memory

The discovered BAR1 corruption makes this a continuing risk. A CPU readback through PRAMIN proves the final VRAM bytes visible through that aperture, but it does not automatically prove that FECS, GPCCS and PGRAPH observe the same translation and cache state.

### Lower probability: packet, subchannel or class decoding

The packet has been consumed and produces distributed PGRAPH/MP traps rather than an obvious front-end-only error. Nevertheless, a controlled invalid-class comparison is needed before closing this branch.

### Very low probability: missing RAMHT entry

The relevant GK104 GPFIFO code attaches engine contexts through the channel instance and supports hardware GR classes directly. Implementing an NV50-style channel RAMHT now would likely add another uncontrolled variable rather than fix the current boundary.

# Phase 1: Produce a register-accurate trap record

The existing statement “PGRAPH trap `0x01000000`, GPC bits `0x0e`, MP trap” is not yet sufficiently specific. Each value must be tied to its exact register.

This distinction matters because Nouveau contains several different, register-specific decoders:

* A top-level PGRAPH trap bit has one meaning.
* A SKED subtrap bit has another.
* A GPC summary value is a bitmap of subordinate units.
* An MP warp error code is an enumerated error reason.

For example, Nouveau labels `0x01000000` as `MEMORY_WINDOW_OVERLAP` in the **SKED error register**. That name must not be applied to a top-level `PGRAPH_TRAP = 0x01000000` unless the SKED status register itself reports that bit. Similarly, a GPC summary value of `0x0e` must not be decoded as MP warp error code `0x0e`; the actual MP warp-error register must contain `0x0e`.

## Required snapshot

Capture the following at four points:

1. Immediately before runlist submission.
2. After `GP_GET` shows the entry consumed.
3. Immediately after the first trap is observed, before clearing anything.
4. After the Nouveau-equivalent W1C clear sequence.

At minimum, record:

```text
PGRAPH_INTR
PGRAPH_TRAP
PGRAPH_GPC_TRAP
PGRAPH_STATUS
PGRAPH_TRAPPED_ADDR
PGRAPH_TRAPPED_DATA

DISPATCH trap/status
CCACHE trap/status
SHADER trap/status
SKED trap/status

For every GPC:
    GPC trap summary

For every TPC in every GPC:
    TPC trap summary
    MP warp-error register
    MP global-error register
```

Also capture FECS and GPCCS state:

```text
FECS CPU control/status
FECS current PC
FECS mailbox/error registers
FECS current channel/context address
FECS next channel/context address
FECS context-memory command/base/target state

GPCCS equivalents where available
```

Store the result as structured JSON rather than only formatted log text:

```json
{
  "test": "set_object_a0c0",
  "class": "0xa0c0",
  "subchannel": 1,
  "pgraph": {},
  "fecs": {},
  "gpccs": {},
  "gpcs": [
    {
      "gpc": 0,
      "summary": "0x...",
      "tpcs": []
    }
  ]
}
```

## Decision from Phase 1

* `DISPATCH CLASS_SUBCH_MISMATCH` or an equivalent front-end error:
  investigate packet encoding, subchannel state and class selection.
* FECS channel/context state changes before the trap:
  the first method is triggering context activation; concentrate on context attachment and restore.
* Identical MP failures in corresponding TPCs:
  concentrate on topology replication, context packs and class-specific GR state.
* FECS or GPCCS memory/translation error:
  concentrate on the context VMA, page tables, MMIO-list VMA and PTE attributes.

# Phase 2: Distinguish class recognition from context activation

Run a small comparison matrix. Each row must start from a full clean hardware reset, not merely cleared trap latches.

| Test | First GR command                                       | Purpose                                                        |
| ---- | ------------------------------------------------------ | -------------------------------------------------------------- |
| A    | None; host/PBDMA-only completion                       | Prove channel and push path remain healthy                     |
| B    | `SET_OBJECT 0xA0C0`, subchannel 1                      | Current failure                                                |
| C    | `SET_OBJECT 0xA0C0`, subchannel 0                      | Exclude subchannel-specific corruption                         |
| D    | `SET_OBJECT 0xA0C0`, subchannel 7                      | Second subchannel control                                      |
| E    | `SET_OBJECT` with intentionally invalid class          | Obtain genuine invalid-class signature                         |
| F    | `SET_OBJECT` with one valid non-compute GK104 GR class | Separate general GR-context failure from compute-class failure |

Choose test F from the classes listed by GK104 Nouveau, such as one of its supported 2D, inline-to-memory or 3D classes; take the exact numeric class from Nouveau’s class definitions rather than guessing it. Nouveau’s GK104 GR class list includes several non-compute controls in addition to `KEPLER_COMPUTE_A`.

## Interpretation

### Invalid class differs; every valid GR class gives the same MP trap

This is strong evidence that the class value is decoded correctly and the shared GR channel context fails during activation.

### Invalid class and A0C0 have exactly the same front-end trap

Investigate the method header, subchannel encoding and front-end class state before context contents.

### Non-compute class binds, but A0C0 traps

Concentrate on compute-specific context state:

* GK104 compute method-pack initialization;
* compute class-specific context image fields;
* shader/MP state;
* method initialization bundles;
* compute-specific MMIO or ICMD packs.

### Trap follows TPC1 regardless of class or subchannel

Concentrate on GR topology and per-TPC context construction.

# Phase 3: Verify the GR engine-context attachment

Add a mandatory `verify_gr_binding()` function and call it immediately before making the channel runnable.

## 3.1 Validate RAMIN `0x210/0x214`

Read these values back through the verified PRAMIN path:

```text
actual_lo = RAMIN[0x210]
actual_hi = RAMIN[0x214]

expected_lo = low32(runtime_context_va) | 0x4
expected_hi = high32(runtime_context_va)
```

Assert exact equality.

Also assert:

```text
runtime_context_va is correctly aligned
runtime_context_va is inside the channel VM
runtime_context_va does not overlap another allocation
the complete 0x2c400-byte range is mapped
```

Do not accept a framebuffer physical address at this location.

## 3.2 Perform a software page walk

Starting from the channel’s PDB recorded in RAMIN, walk every page covering:

* runtime GR context;
* per-channel MMIO list;
* pagepool;
* bundle circular buffer;
* attribute buffer;
* LTC-related allocation;
* page tables themselves.

For every mapping, print:

```text
VA
PDE address/value
PTE address/value
resolved framebuffer address
target type
read/write state
privileged state
page size
```

Reject:

* unmapped pages;
* mixed page sizes not intended by the allocator;
* system-memory target for a VRAM object;
* absent privileged bit where Nouveau requires it;
* physical discontinuities not represented in the page table;
* mapping through a different channel VM.

## 3.3 Verify FECS sees the intended channel

At the failing submission, compare:

```text
channel RAMIN instance address
RAMFC instance programmed into the runlist
FECS current/next channel address
GR context VA from RAMIN + 0x210
```

A mismatch here would mean the correct context was built but the scheduler or FECS selected a different channel instance.

## Likely fixes from Phase 3

* Rewrite `0x210/0x214` with the mapped channel VA, not VRAM physical address.
* Preserve the low control bits exactly, including `| 0x4`.
* Correct the PDB used by the channel.
* Add the missing privileged PTE flag.
* Flush page-table and LTC state after the final modification and before runlist submission.
* Rebuild the runlist only after all RAMIN and mapping writes are complete.

# Phase 4: Reproduce `gf100_gr_chan_bind()` byte for byte

Build a dedicated serializer corresponding to Nouveau’s binding operation rather than patching the context in scattered code.

The sequence should be:

```text
1. Allocate the runtime GR-context object.
2. Map it in the channel VM.
3. Copy exactly 0x2c400 bytes from the verified golden context.
4. Build and map the per-channel MMIO list.
5. Patch the runtime-context header.
6. Attach the runtime-context VMA at RAMIN + 0x210.
7. Flush and invalidate the required memory/translation state.
8. Verify all bytes and mappings.
9. Only then submit the runlist.
```

For the current 24-entry MMIO list, the non-secure Nouveau-style context header should resolve to:

```text
context + 0x00 = 24
context + 0x04 = mmio_list_va >> 8
```

This follows Nouveau’s use of `mmio_nr / 2`, where `mmio_nr` counts address/value dwords rather than logical register entries. Confirm that the current builder is not accidentally writing `48`, bytes instead of entries, or a physical address instead of `VA >> 8`.

Add these assertions:

```text
context_header_count == actual logical entry count
context_header_list_va << 8 == actual mapped list VA
list VA is 256-byte aligned
all list entries are complete address/value pairs
list allocation is mapped privileged where required
```

Dump the complete 24-entry list in execution order:

```text
index
register
value
referenced resource
resource VA
resolved physical address
PTE attributes
```

# Phase 5: Audit golden-context generation for topology errors

The trap distribution makes this the highest-value code audit.

## 5.1 Verify detected topology before context generation

Record and assert:

```text
GPC count = 4
total TPC count = 8
TPC count per GPC = [2, 2, 2, 2]
```

Read these values both:

* immediately before `grctx_main`;
* immediately before FECS saves the golden context.

A reset, power-gating transition or partially initialized topology register between those stages could make the context generator cover fewer units than the active hardware later exposes.

## 5.2 Audit every GPC/TPC replication loop

For every register pack classified as:

* global;
* GPC;
* TPC;
* PPC;
* ROP/LTC;
* method initialization;
* ICMD initialization;

log the expanded destination addresses.

For four GPCs and two TPCs per GPC, every TPC-scoped entry should produce eight writes. Assert this programmatically.

Specifically inspect:

```text
GPC base stride
TPC base stride inside each GPC
loop bounds
per-GPC TPC count lookup
broadcast versus unicast registers
address-mask application
pack count interpretation
```

The `TPC1 in GPC1–3` pattern is compatible with:

* starting a GPC loop at the wrong index;
* applying TPC1 initialization only in GPC0;
* using a GPC stride where a TPC stride is required;
* incrementing both GPC and TPC offsets for the same loop;
* decoding a context-pack repetition count incorrectly.

## 5.3 Verify class-specific method and ICMD packs

The compute class may depend on state installed through GR method or ICMD initialization during golden-context construction.

Compare the exact expanded sequence against Nouveau’s GK104 context-generation source:

```text
pack order
register addresses
values
wait-for-idle points
broadcast masks
GPC/TPC expansion
method class and method address
```

Do not merely compare the static tables. Compare the final expanded write stream produced by `add.py`.

A useful artifact is:

```text
expanded_nvgpu_context_writes.txt
expanded_nouveau_expected_writes.txt
```

Then diff by:

```text
sequence number
address
value
scope
GPC
TPC
```

## 5.4 Inspect live per-TPC state before golden save

Immediately after `grctx_main`, but before asking FECS to save the golden context, read representative contexted registers from:

```text
GPC0/TPC0
GPC0/TPC1
GPC1/TPC0
GPC1/TPC1
GPC2/TPC0
GPC2/TPC1
GPC3/TPC0
GPC3/TPC1
```

A missing or default-valued state in the same units that later trap would locate the error before FECS save/restore.

## Likely fix

Correct the expansion logic or missing context pack, regenerate the golden context, and require that all eight TPC instances show initialized live state before accepting the golden image.

# Phase 6: Isolate the per-channel MMIO list

Once the list format and mapping have been verified, use controlled prefix tests to locate a bad entry.

Every test must begin with a cold reset because an incomplete context restore may leave GR in an unusable state.

Test:

```text
prefix length 0
prefix length 12
prefix length 18
prefix length 21
prefix length 24
```

This is diagnostic only; a zero or partial list is not a valid final configuration.

For each prefix, record:

```text
Does SET_OBJECT trap?
Which GPC/TPC units trap?
Does FECS report a translation error?
Does the trap signature move?
```

## Interpretation

* Prefix `0` avoids the MP trap:
  one or more MMIO entries, their referenced resources or their mappings are responsible.
* The first failing prefix identifies the first suspect entry.
* Every prefix, including `0`, gives the same trap:
  the failure is likely in the base golden context or GR-context attachment.
* A shortened list changes the failure into an FECS memory fault:
  the list is being consumed, and the original MP trap likely occurs after at least some channel patching.

After locating a boundary, inspect the first added entry and its target allocation:

```text
entry register
entry value
resource base
resource size
resource alignment
resource PTE
privileged bit
zero/init contents
```

# Phase 7: Prove GPU-side visibility

Because bulk BAR1 writes were demonstrably corrupt, use the verified PRAMIN dword path for all context-critical final writes:

```text
channel RAMIN
RAMFC
PDB/PDE/PTE pages
runtime GR context
MMIO list
pagepool
bundle buffer
attribute buffer
runlist
GPFIFO
SET_OBJECT push
```

For every object:

1. Write it.
2. Read it back.
3. Repair discrepancies.
4. Read it back again.
5. Record a complete hash.
6. Perform the required cache/TLB invalidations.
7. Read key words again immediately before submission.

Record both:

```text
hash after construction
hash immediately before runlist submission
```

If they differ, identify which initialization step overwrote the object.

Also capture FECS context-memory base/target/command registers while the first context load is occurring. This can reveal that FECS is reading:

* the wrong target;
* the wrong base;
* a stale address;
* a different context instance.

The clean pre-submit trap latches prove that the reported trap is new. They do not prove that the saved context or GPU-visible memory is clean.

# Phase 8: Compare against a real Nouveau channel

The most decisive reference is the same GK104 board running Linux Nouveau.

Instrument Nouveau immediately after it creates a minimal GK104 compute channel and before its first compute method. Dump:

```text
channel instance 0x000–0x2ff
RAMFC fields
RAMIN 0x210/0x214
channel PDB
GR-context VMA
full GR-context hash
first 0x100 bytes of GR context
full per-channel MMIO list
MMIO-list VMA and PTEs
pagepool/bundle/attribute VAs and PTEs
relevant FECS/GPCCS state
expanded context-generation writes if practical
```

Then submit only:

```text
SET_OBJECT 0xA0C0
```

and capture the same trap/state snapshot used by the bare-metal implementation.

The comparison should be normalized into this table:

| Field                  | Nouveau | nvgpu | Match |
| ---------------------- | ------: | ----: | ----- |
| RAMIN `0x210`          |         |       |       |
| RAMIN `0x214`          |         |       |       |
| Context header count   |         |       |       |
| Context header list VA |         |       |       |
| MMIO list entries      |         |       |       |
| Context mapping PTE    |         |       |       |
| MMIO-list mapping PTE  |         |       |       |
| FECS selected channel  |         |       |       |
| First-method trap      |    zero |       |       |

This is more reliable than inferring undocumented fields one by one.

# Phase 9: Verify GR initialization and power state

A GR method can be the first point that exposes an incomplete engine initialization even when FECS golden-context generation appeared to succeed.

Before and immediately after `SET_OBJECT`, verify:

```text
PGRAPH master enabled
GR engine not auto-gated unexpectedly
FECS running and idle
GPCCS running and idle
GPC/TPC topology still visible
GR interrupt and trap latches clear
context-switch engine not busy from golden generation
runlist channel ID matches FECS-selected channel
```

Compare the complete ordering of:

```text
reset release
clock enable
PGRAPH master enable
global GR MMIO packs
FECS/GPCCS firmware load
golden-context generation
golden-context save
runtime-context binding
runlist submission
```

A successful firmware startup does not prove that every static GR initialization pack required for later context restoration was executed.

# Validation ladder after the fix

Do not return immediately to the complete vector-add launch. Reintroduce operations one boundary at a time.

## Stage 1

```text
SET_OBJECT A0C0
```

Success criteria:

```text
PBDMA consumes push
no PGRAPH trap
no GPC/TPC/MP trap
FECS becomes idle
```

## Stage 2

```text
SET_OBJECT
shader-cache/invalidation method
```

## Stage 3

```text
SET_OBJECT
code-address methods
```

## Stage 4

```text
SET_OBJECT
QMD/launch-descriptor address
```

## Stage 5

```text
SET_OBJECT
empty-kernel LAUNCH
semaphore release
```

Success criteria:

```text
semaphore changes
empty kernel exits
no trap
```

## Stage 6

```text
genuine SM30 vector-add kernel
```

Final acceptance criteria:

```text
hardware_demo=ok
correct numerical result
all trap registers remain zero
no repaired corruption after submission
context can be switched away and restored again
second launch succeeds without rebuilding the channel
```

The second launch and context switch are essential. A single successful launch could conceal a context that works only while resident but cannot be saved and restored.
